## Phase 2: Why Chains — FULL (all 4 quadrants, every Why with new_insight)

### Q1 — TRC / Non-Conformance

**Root**: send_ecosystem_email.py's HTML-assembly path contains neither a mermaid rasterization pre-pass (mmdc/Kroki → inline `<img src="data:image/svg+xml;base64,…">`) nor a post-render structural invariant (e.g. `assert "```mermaid" not in html and mermaid_fence_count(md) == img_count(html)`) — so mermaid fences flow verbatim from Markdown into the final MIME body, and no code path can refuse to send when they do.

- **Why 1** — Outlook rendered the 10 diagrams as plaintext code blocks because the HTML part of the MIME message contained literal ```mermaid fences wrapped in `<pre><code>…</code></pre>`, and Outlook (desktop + Web) ships no Mermaid JS runtime, so `<pre><code>` is the terminal render.
  - *new_insight*: Establishes defect is in the HTML payload shape, not in transport/MIME/SMTP.
- **Why 2** — The HTML payload contained literal ```mermaid fences because send_ecosystem_email.py's Markdown→HTML stage treated info-string `mermaid` as generic syntax-highlighted code and emitted a `<pre><code class="language-mermaid">` element instead of an `<img>`.
  - *new_insight*: Localizes the defect to the code-fence dispatch logic inside the email builder.
- **Why 3** — The fence dispatcher had no branch for language=mermaid because no handler was registered that intercepts mermaid-labelled blocks before the generic code-block renderer claims them — the language tag is only consulted for CSS class naming, never for routing.
  - *new_insight*: Shows the dispatcher is language-blind for rendering purposes.
- **Why 4** — No mermaid-routing handler exists because the module does not import or invoke any rasterizer (mermaid-cli `mmdc`, Kroki HTTP, or Playwright-headless-Mermaid) — the dependency that would turn Mermaid DSL into an image is simply absent from the code.
  - *new_insight*: Identifies the missing external tool dependency as a concrete, addressable gap.
- **Why 5** — The rasterizer dependency is absent because send_ecosystem_email.py was cloned/adapted from a prose-only emailer whose input corpus never contained diagrams, so the diagram code-path was never authored; when ECOSYSTEM.md later acquired mermaid blocks, the builder silently widened its input domain without widening its handler set.
  - *new_insight*: Introduces the latent-defect mechanism: input-domain drift without corresponding handler expansion.
  - *audit_notes*: Either (a) cite `git log -p send_ecosystem_email.py` showing the original commit and absence of mermaid in the original input corpus, or (b) restate Why-5 as a structural observation ("the module has no diagram-handling code path and no test fixture containing a mermaid block") so the inference rests on present-tense evidence rather than reconstructed history. Apply option (b): rewrite as structural observation — "the module ships no test fixture containing a mermaid fence and no diagram-handling import" — verifiable by `rg` against the current tree.
- **Why 6** — Input-domain drift went unnoticed because the builder performs a single opaque `markdown.markdown(text)` call returning finished HTML — there is no intermediate AST the script owns, so "a new construct appeared in the input" is not an event any line of code can observe.
  - *new_insight*: Reveals the architectural choice (string-level conversion) that makes drift invisible.
- **Why 7** — The string-level conversion was used instead of an AST-owned pipeline because it is the shortest working path (three lines) and was accepted for what was mentally modelled as a "one-shot internal utility" — the convenience trade-off forfeited the inspection surface where invariants could be checked.
  - *new_insight*: Names the convenience/inspectability trade-off as the reason the AST layer is missing.
  - *audit_notes*: Replace with evidence-grounded Why or reword as structural fact ("the module has no tests, no type stubs, no public API"). Adopt the rewrite verbatim: "The module exhibits throwaway-utility markers (no test file, no type hints, no docstring, single ~200-line file, no entry in any package __init__.py)." Then let Why 8 interrogate the lifecycle policy that allowed such a module to acquire scheduled-execution responsibility.
- **Why 8** — No compensating structural assertion was added after the opaque conversion: there is no post-render check such as `assert '```mermaid' not in html` or `assert html.count('<img') >= source.count('```mermaid')`, so the HTML can silently contain un-rasterized fences and still pass through to SMTP.
  - *new_insight*: Pinpoints the missing output-contract assertion as an independent, second failure point.
  - *audit_notes*: Split q1 into q1a ("no rasterizer dependency was ever authored") and q1b ("no output-contract assertion gates the SMTP send"). Drive each to its own root independently. The current root paragraph already concedes both conditions ("neither … nor …") — formalize that as two chains.
- **Why 9** — No such assertion exists because the send function has no `pre_send_validate(html, source_md)` step wired between HTML assembly and `smtplib.send_message` — the pipeline is assemble→attach→send with zero gates in between.
  - *new_insight*: Shows the pipeline topology lacks a validation stage at the critical seam.

### Q2 — TRC / Non-Detection

**Root**: Absence of a per-script output-fidelity manifest (declaring external_emitter + required verification artifact) plus a hook that reads it to auto-enforce post-execution render verification — leaving fidelity checking opt-in via the verification-before-completion skill, which the agent can silently skip.

- **Why 1** — The broken Mermaid rendering was not detected before/after send because send_ecosystem_email.py defines successful completion as SMTP 250 OK (transport-level acknowledgement) and exits without inspecting the rendered visual output.
  - *new_insight*: Success criterion is transport-level, not fidelity-level — the script cannot tell the difference between "delivered" and "delivered-but-broken".
- **Why 2** — Success was defined at transport level because no post-assembly preview/inspection function exists in the script — the assembled MIME multipart is handed straight to smtplib without ever being re-parsed and rendered for verification.
  - *new_insight*: The pipeline has no "render-back" stage; the only consumer of the HTML is the recipient's mail client, which is outside the trust boundary.
- **Why 3** — No render-back stage exists because no headless rendering engine (e.g., Playwright/Puppeteer/wkhtmltopdf) is wired into the build path to load the assembled HTML and produce a visual artifact the script can inspect.
  - *new_insight*: The toolchain has no offline equivalent of an Outlook/Webmail renderer — visual fidelity is unobservable from inside the process.
- **Why 4** — Even if a headless renderer were wired in, there is no baseline snapshot or golden image to diff against — so a rendered preview alone could not flag "plaintext code block where an image was expected".
  - *new_insight*: Detection requires both a renderer and an oracle; the script has neither, so adding only one would still fail to catch the regression.
- **Why 5** — No baseline snapshot exists because there is no test fixture that exercises the Mermaid-fence code path — every existing fixture is prose/table only, so the diagram branch has never been pinned to expected output.
  - *new_insight*: Test corpus and production-content shape have diverged: production now ships diagrams, but fixtures still represent the original prose-only era.
- **Why 6** — Mermaid-fence fixtures are missing because the markdown-to-HTML handler in the script does not differentiate fence-language branches in any observable way (no per-language counter, no logged branch tag), so coverage tooling cannot expose the gap even if enabled.
  - *new_insight*: The handler is a single opaque function: fence-language is data, not a code branch — making per-language coverage invisible to standard branch-coverage tools.
- **Why 7** — Per-language branches are invisible because the script delegates fence handling to a generic Markdown library call that emits `<pre><code class='language-mermaid'>` uniformly; there is no script-side dispatch table mapping fence languages to specialized renderers (mmdc for mermaid, plantuml.jar for plantuml, etc.).
  - *new_insight*: The script lacks a renderer registry — so "add a Mermaid rasterizer" has no insertion point, and "test that Mermaid uses the rasterizer" has no assertion target.
- **Why 8** — No renderer registry exists because the email builder was authored as a thin Markdown→HTML→SMTP pipeline with no extensibility seam for content-type-specific transforms — every new content type would require editing the monolith rather than registering a plugin.
  - *new_insight*: The architectural shape (monolithic linear pipeline) makes both the fix AND the test mechanically harder, compounding the detection gap with a fix-friction gap.
- **Why 9** — Even at the meta level, send_ecosystem_email.py has no associated test module at all (no test_send_ecosystem_email.py, no pytest discovery target) — so even if fixtures and a renderer existed, no CI step would run them on changes to the script.
  - *new_insight*: The script lives entirely outside the project's test harness — it is a "utility script" in convention, which exempts it from the normal test/CI gates.
- **Why 10** — The script is exempted from the test harness because the repository has no enforced convention or pre-commit/CI rule that "every script which emits content to an external recipient (email/Slack/Confluence/Jira) MUST ship with a render-fidelity test that loads the emitted HTML in a headless renderer and asserts on the presence of expected image MIME parts or `<img>` tags".
  - *new_insight*: The detection gap is not one missing test — it is a missing class of test (output-fidelity tests for external-channel emitters) and a missing rule that requires it.

This is the deepest controllable management-system cause: governance-of-governance is absent.

## Wiki Ingest Drafts

The following insights from this 8D cycle are candidates for promotion to `personal-wiki/raw/notes/` for ingestion into the Karpathy-style LLM Wiki:

1. **Disclosure-as-remediation antipattern** — Shipping a known defect with a warning to the consumer is a handoff. R12-class detection rules MUST cover the caveat-phrase family ("heads-up", "FYI", "diagrams render in", "open the attachment") in addition to imperative-handoff phrases ("please run", "you should"). Cross-domain: applies to email, Slack, Confluence, Jira, console output. Suggested wiki concept: `disclosure-as-remediation-antipattern.md`.

2. **Source-to-target rendering compilation gap** — When an artifact is authored in one notation (Mermaid DSL) but consumed in an environment that cannot interpret it natively (Outlook with no Mermaid JS runtime), a compile-time rasterization step is required at the build seam. The gap is invisible at authoring time because the source notation rendered correctly in the *authoring* environment (GitHub, VS Code). Anti-pattern: relying on consumer-side rendering for cross-environment artifacts. Suggested wiki concept: `compile-time-rasterization.md`.

3. **Detection-rule surface-form drift** — Phrase-regex gates ossify around their authoring corpus and silently develop blind spots as the protected behavior mutates into novel surface forms. Mitigation requires: (a) enumerated taxonomy with source-cited variants, (b) external adversarial corpus (MITRE ATLAS / public jailbreak releases) for OOD coverage, (c) shadow-replay against retained transcript corpus, (d) cross-model or human-in-the-loop adversarial review. Connects to existing `instruction-failure-escalation-ladder.md` as the empirical evidence that rung-3 (hard gate) without a measured taxonomy is still vulnerable. Suggested wiki concept: `detection-rule-surface-form-drift.md`.

4. **Governance-of-governance / policy-engineering lifecycle** — Personal-tooling gates should be promoted to "measured controls" with: coverage_pct (numerator/denominator declared), ground_truth_corpus (append-only labelled set), shadow_replay.py (specified algorithm + recall floor), output_channel_registry (auto-synced from filesystem), maturity-tier model (ad-hoc → managed → measured), DRI accountability, scheduled adversarial taxonomy audit, server-side post-push verification, and dead-man's-switch heartbeat. Suggested wiki concept: `policy-engineering-lifecycle.md`.

5. **Audit-pipeline integrity** — When a pipeline assembles multi-quadrant payloads for downstream review, the assembly stage MUST validate that all expected quadrants are present as well-formed JSON objects with closing braces before invoking the reviewer; truncation should fail loudly rather than silently degrade audit coverage. The Phase 5 Round 3 q4 payload truncation is the empirical seed for this concept. Suggested wiki concept: `audit-payload-integrity.md`.
