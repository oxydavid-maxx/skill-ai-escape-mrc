# 8D Report: Gate-Pipeline-Enforcement Gap (Generative Class)

**Date**: 2026-04-25T15:50:57.944361
**Problem**: GATE-PIPELINE-ENFORCEMENT GAP. The policy engine has multiple failure modes that let the agent ship substantial code without going through the brainstorm-plan-execute pipeline: (1) R4 only fires when a sizable-task banner is active in the transcript, which itself depends on UserPromptSubmit's keyword regex matching the user's specific prompt — many real-world prompts ('go', 'do it', 'no action from those reports?', 'fix this', 'why X?') do not match, so R4 stays latent and agent freely Writes/Edits substantial code. (2) R2 fires on spec writing when brainstorming wasn't invoked since sizable_task_ts — but R2's transcript-scan predicate apparently does NOT detect Skill tool invocations correctly: this session invoked superpowers:brainstorming explicitly via Skill() tool, then attempted to Write the spec, and R2 still blocked claiming brainstorming wasn't invoked. Either the predicate's regex/structure for detecting Skill invocations is wrong, or there's a timing/caching issue between Skill call and PreToolUse Write evaluation. (3) Even when banners DO inject correctly and skills ARE invoked, R9 (banner-required skill not called) is the only enforcement; once R9 is satisfied (skill invoked even briefly), nothing prevents the agent from proceeding to execute Write/Edit/Bash without the FULL pipeline (writing-plans -> plan -> execute). (4) The execute-prevention-actions rule I added today expanded the keyword set, but doesn't structurally close the class — every novel-phrasing prompt is a potential bypass. Run full ecosystem 8D on the GENERATIVE class of pipeline-enforcement gaps. Goal: prevention design that doesn't rely on keyword-matching the user's prompt at all, but instead detects 'agent is about to ship substantial code' from the AGENT's own behavior (number of unique files touched in last N tool calls, lines-changed, file types, etc.) and forces the pipeline regardless of what the user said. Plus: identify why R2's brainstorming-detection predicate fails despite the Skill being invoked. Heartbeat is built-in (5-min ticks).

## Phase 2: Why Chains (4 quadrants — FULL)

### Q1 (TRC × Non-Conformance)

**Root**: The R2 predicate's transcript-scan logic uses a single regex/string-match heuristic over a serialized transcript view that does not include Skill tool invocation records as parseable structured events, causing it to miss any in-session `Skill(superpowers:brainstorming)` call regardless of phrasing — a fundamentally wrong data source for the question being asked.

1. **Why 1**: The agent successfully executed Write/Edit operations to ship substantial code without first traversing the brainstorm→plan→execute pipeline, even though R2/R3/R4/R5 were configured in enforce mode to block exactly this. *(Audit notes: replace with the actual first cause; treat as problem-statement header outside the chain — Round 1, Round 2, Round 3 all flagged this.)*
2. **Why 2**: PreToolUse evaluation of Write/Edit did not deny the call: either no sizable_task banner was active (R2/R3/R4/R5 stayed dormant), or a banner was active but R2's brainstorming-already-invoked predicate falsely returned true/false in a way that let the call through. *(new_insight: Two distinct failure surfaces — banner-absent vs predicate-misfire — both produce the same observable bypass.)* *(Audit notes: split q1 into q1a [banner-absent / arming failure] and q1b [predicate-misfire / detection failure] — Rounds 1, 2, 3 all flagged this fork.)*
3. **Why 3**: The sizable_task banner is emitted by the UserPromptSubmit hook only when the user's prompt text matches a hard-coded keyword regex (e.g., "implement", "build", "design", "add feature"); short imperative prompts like "go", "do it", "fix this", "why X?" do not match, so the banner is never injected and the rules that depend on it never arm. *(new_insight: arming signal for the entire process-skip rule family is coupled to surface-form lexical features of user input.)*
4. **Why 4**: The keyword regex is the sole trigger basis because the gate-rules.yaml schema only supports prompt-text predicates at UserPromptSubmit time — there is no available predicate type that inspects the agent's downstream tool-call trajectory (files touched, LOC about to be written, tool-call cardinality) to retroactively arm the banner. *(new_insight: rule engine's predicate vocabulary is exclusively input-side; agent-behavior signals are not first-class.)*
5. **Why 5**: When the banner DOES fire and brainstorming IS invoked via `Skill(superpowers:brainstorming)`, R2's predicate that checks "was brainstorming invoked since sizable_task_ts" still returns "not invoked" and blocks the legitimate spec write — because the predicate scans the rendered transcript for a textual marker that the Skill tool's actual invocation record does not produce in the form the regex expects.
6. **Why 6**: The R2 predicate was authored assuming Skill invocations would appear in the transcript as a literal string like "superpowers:brainstorming" inside a recognizable tool_use block, but Skill tool calls are serialized as a structured tool_use entry whose `name` field is `Skill` and whose `input.skill` field carries the skill name — a single naive regex over the flat transcript view either misses the nested field or matches partial strings inconsistently. *(new_insight: the predicate confuses surface text with structured event data; the right query is over `tool_use.input.skill`, not over a flat string.)*
7. **Why 7**: Only one detector exists per rule (R2 has a single predicate with no fallback), so when that predicate's data source is wrong there is no second corroborating signal (e.g., file-system marker that brainstorming wrote a spec stub, or session-state flag set by a SubagentStop hook) to overrule the false negative. *(new_insight: predicates are unary, not ensembles — single point of failure per rule.)* *(Audit notes: relocate to q3_mrc_nc — this is a management-system / detection-gap defect, not a TRC depth step. Rounds 1 and 3 flagged.)*
8. **Why 8**: Even if R2 fired correctly, R9 ("banner-required skill not called") is satisfied by ANY single invocation of the named skill; once `Skill(superpowers:brainstorming)` returns, R9 stops blocking and the agent can proceed straight to Write/Edit without ever invoking writing-plans or executing-plans — because no rule encodes the SEQUENCE invariant (brainstorm → plan → execute must all occur before substantial code). *(new_insight: rules check membership but not ordering; the pipeline is a sequence yet enforcement is set-based.)* *(Audit notes: promote to q1c [ordering-invariant absence] as its own chain — Rounds 1, 2, 3 flagged.)*
9. **Why 9**: The enforcement model treats each pipeline phase as an independent skill-invocation requirement rather than as states in a finite-state machine where Write/Edit is only legal from the post-execute state — there is no session-level state object that tracks "current pipeline phase" and gates tools by phase. *(new_insight: harness lacks a pipeline-state machine; rules try to approximate one with stateless predicates and fail.)*
10. **Why 10**: Today's mitigation (expanding the keyword set in the execute-prevention-actions rule) treats each newly observed bypass phrase as a distinct fix, but the bypass class is generative — natural language has unbounded paraphrases for "do the thing", so any finite keyword list will be incomplete forever; the patch addresses instances, not the class. *(new_insight: keyword-set expansion is asymptotically defeated by paraphrase; the class cannot be closed input-side.)* *(Audit notes: move to CA-matrix SoA findings — this is a critique of a corrective action, not a TRC step. Rounds 1, 2, 3 flagged.)*
11. **Why 11**: All the above defects share a single architectural commitment: the gate decides "is this a sizable task" from the USER'S prompt rather than from the AGENT'S impending action, so the gate has no access to the variable that actually determines whether enforcement is needed (the size and shape of what is about to be written). *(new_insight: trigger basis is mis-coupled to input rather than output; the question "should I enforce the pipeline?" is answered from the wrong variable.)* *(Audit notes: tag explicitly as synthesis node, or replace with a deeper cause [why was the input-side schema chosen originally?]. Rounds 1 and 2 flagged.)*
12. **Why 12**: The trigger-basis mis-coupling persists because no PreToolUse-level aggregator computes rolling agent-behavior metrics (unique files touched in last N tool calls, cumulative LOC delta staged for write, file-type mix, presence of new module/hook/skill paths) to feed predicates — that module simply does not exist in `~/.claude/hooks/`.
13. **Why 13**: It does not exist because R2/R3/R4/R5 were originally drafted around the simplest available signal (user prompt text at UserPromptSubmit time) and never refactored when the failure mode shifted from "user forgot to ask for the pipeline" to "agent shipped code without it" — the rule schema was frozen before the agent-behavior class of bypasses was recognized. *(new_insight: schema freeze preceded threat-model evolution; rules cannot express the predicate they now need.)*
14. **Why 14 (TRC)**: Combined with R2's structured-vs-flat transcript scan bug, the result is a dual technical defect: (a) the data source for "has brainstorming happened" is the wrong shape (flat regex over a structured event stream), and (b) the data source for "is enforcement needed" is the wrong variable (user input keywords vs agent output behavior); fixing either alone leaves the other class open, which is why the deepest controllable cause is the predicate/data-source layer's lack of structured-event access and behavior-side aggregation.

### Q2 (TRC × Non-Detection)

**Root**: No closed-loop conformance test exists that simulates real prompts + tool sequences end-to-end and asserts pipeline-traversal invariants on the resulting transcript, so any predicate divergence between emit-side and consume-side stays invisible until a live session happens to exercise the gap.

1. **Why 1**: The pipeline-enforcement gap (R4 keyword latency + R2 predicate miss + R9 "any invocation satisfies" + keyword-set whack-a-mole) was not detected before today's live session escape because the policy engine ships with no automated test that drives synthetic prompts through UserPromptSubmit + PreToolUse and asserts whether substantial Write/Edit got blocked or allowed.
2. **Why 2**: No such test exists because gate-rules.yaml predicates were authored as single-source regex/transcript-scan logic and validated only by running `claude-hooks lint` (syntactic) — there is no fixture-based behavioral harness that replays a transcript and observes the deny/allow decision. *(new_insight: lint validates rule SHAPE, not rule BEHAVIOR — a predicate can be syntactically valid YAML and still never fire on the inputs it was meant to fire on.)*
3. **Why 3**: A behavioral harness was never built because the trigger-emission layer (UserPromptSubmit banner) and the trigger-consumption layer (PreToolUse predicate scan for prior `Skill()` calls) were authored at different times in different files, so each side was tested in isolation against its own mental model of the transcript shape — never against the SAME transcript. *(new_insight: cross-layer predicate divergence — emitter writes one transcript marker, consumer scans for a different one — is the exact bug class only end-to-end tests catch, invisible to per-layer unit tests.)*
4. **Why 4**: Predicate divergence specifically went undetected for R2's brainstorming-Skill scan because the consumer's regex was written against an assumed Skill-tool transcript signature (likely a plain-text "invoking superpowers:brainstorming" string) while the actual `Skill()` tool produces a structured tool_use block whose serialization the regex never matches. *(new_insight: the R2 predicate is grepping for a textual artifact that the harness never emits — wrong needle, no canary.)*
5. **Why 5**: No "predicate hit-rate canary" exists because metrics.jsonl records deny/allow counts but does not record the COMPLEMENT — predicates that SHOULD have evaluated true but evaluated false (false negatives) are unobservable from current telemetry, since by definition the agent proceeded as if the precondition wasn't met. *(new_insight: telemetry is asymmetric — blocks are visible, missed-blocks are invisible — so a silently-broken predicate produces zero telemetry signal, identical to "rule never needed to fire.")*
