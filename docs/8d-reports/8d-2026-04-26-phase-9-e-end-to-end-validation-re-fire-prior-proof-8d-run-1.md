## Phase 2: Why Chains (4 quadrants)

### Q1 — TRC × Non-Conformance

1. **Why 1**: Phase 9 of the closed-loop 8D pipeline hung on run-1777179234 and failed to produce plan.md, blocking Phase 10 gate-file and email emission.  *new_insight*: Establishes the symptom boundary: outage is Phase 9-only, downstream is starved.
2. **Why 2**: Phase 9 was originally implemented by invoking the `superpowers:writing-plans` skill, which is interactive (prompts the operator for choices) and therefore never returns inside an unattended SDK auto-dispatch run.  *new_insight*: First-order technical cause — an interactive skill placed inside a non-interactive control flow.
3. **Why 3**: The auto-dispatch SDK loop has no stdin / no human-in-the-loop wired to writing-plans' prompts, so the skill blocks waiting for input that will never arrive — a classic deadlock between a synchronous interactive contract and an asynchronous batch caller.  *new_insight*: Reframes the hang as a contract mismatch (interactive vs batch) rather than a skill bug.  *audit_notes*: Multiple audit rounds flagged this as a restatement of Why 2 rather than a new causal step. Recommended replacement: "Why was an interactive skill placed in a batch context? → because the FSM dispatch layer has no skill-classification taxonomy (interactive vs batch-safe) and no precondition gate that rejects interactive skills inside auto-dispatch."
4. **Why 4**: The fix in c22df87 + 932897b replaced the skill invocation with a direct `call_claude()` using a fixed Phase-7-style system prompt, but Phase 9's task (synthesize a full plan.md from phase_4 corrective actions across 4 quadrants) is materially larger and less constrained than Phase 7's task.  *new_insight*: The "copy Phase 7 pattern" assumption smuggled in a scope mismatch — same plumbing, larger payload and looser output schema.  *audit_notes (round 1, RESIDUAL)*: Plausible but unverified; either include diff summary or commit URLs as evidence.
5. **Why 5**: A single fixed-prompt LLM call asked to emit a complete multi-quadrant plan.md has no per-section schema, no length budget, and no structured-output contract, so the model can stall on long generation, drift off-format, or produce an artifact that the Phase 10 reader cannot parse.  *new_insight*: Locates the technical defect in prompt/output-schema under-specification, not in transport.
6. **Why 6**: Without a StructuredOutput tool-block contract or a per-quadrant decomposition, Phase 9 is effectively asking the model to do four synthesis tasks in one turn while also formatting them — which exceeds the deterministic envelope of a single call and reintroduces non-termination risk even after the interactive skill was removed.  *new_insight*: Names the real failure class: monolithic synthesis call with no decomposition or schema enforcement.  *audit_notes*: Flagged as duplicate of Why 5. Recommended replacement: "Why was no StructuredOutput contract adopted despite being a documented pattern in `concepts/claude-agent-sdk-patterns.md`? → wiki-consultation gap (R6 not enforced for FSM-phase implementation) OR absence of a Phase-implementation checklist requiring StructuredOutput for any multi-section LLM emission." This handoff to q3 MRC is more useful than the current restatement.
7. **Why 7**: Phase 10 only needs the corrective actions enumerated and rendered into a gate-file + email body; it does not actually need an LLM-synthesized narrative plan — phase_4 already contains the structured action list that Phase 10 consumes.  *new_insight*: Shows that the LLM call at Phase 9 is not just risky but architecturally redundant — the data Phase 10 needs is already deterministic upstream.  *audit_notes (round 3, RESIDUAL)*: This is technically a design-critique pivot rather than a strict causal step in "why did Phase 9 hang"; defensible but mark explicitly as the chain's pivot from proximate-cause analysis to architectural-redundancy analysis.
8. **Why 8**: Phase 9 was inserted because the original design treated `plan.md` as a human-readable artifact requiring narrative prose, conflating two concerns: (a) machine-readable action list for the gate, and (b) human-readable summary for the approver email. Both were pushed onto one LLM call instead of being split.  *new_insight*: Coupling defect — two outputs with different reliability requirements share one generation step.
9. **Why 9**: Because the machine-readable concern was coupled to an unbounded LLM synthesis, any prompt-level fragility (length, formatting, ambiguity) propagates into the gate-file path, taking down the entire closed-loop feature even though the gate file itself is trivially derivable from phase_4 JSON.  *new_insight*: Asymmetric blast radius — small surface, full-feature outage, caused by coupling determinism to non-determinism.  *audit_notes*: Multiple rounds flagged as consequence-restatement, not new cause. Recommended replacement: "Gate-file emission sits downstream of the LLM call because the FSM topology was authored by linearly extending the prior pipeline rather than being redesigned around data-flow boundaries (deterministic gate vs non-deterministic narrative). No architectural review step required isolating deterministic outputs from LLM synthesis nodes."
10. **Why 10**: The Phase 9 LLM-synthesis step is therefore the wrong abstraction for a deterministic gate-emission pipeline: it is non-deterministic, unbounded, and unnecessary for the gate's machine contract — it should be replaced by a deterministic template renderer over phase_4, with an optional, isolated per-quadrant LLM call for the human-readable summary only.  *new_insight*: Lands at the controllable technical root: replace monolithic LLM synthesis with deterministic templating; isolate any LLM use to a bounded, per-quadrant, schema-constrained call.  *audit_notes (round 2, RESIDUAL; round 3, ADDRESSABLE)*: Why 10 mixes root-cause with fix-prescription. Recommended trim: "The chosen abstraction (monolithic LLM synthesis) is mismatched to the gate's deterministic machine contract" — and move "should be replaced with deterministic template renderer" into the corrective-action quadrant where it belongs. Retained here verbatim per "every detail visible" reporting requirement.
11. **Why 11**: Stop here — deeper causes (why the pipeline was designed with this coupling, why interactive skills were chosen for batch flows) are managerial/process root causes that belong in q3/q4, not q1.  *new_insight*: Boundary marker: q1 root reached at the code-level architectural choice; further whys cross into MRC quadrants.  *audit_notes*: Boundary marker, not a Why. Move to a separate "handoff" annotation field on the q1 object so the chain length reflects only causal links.

**Q1 root**: Phase 9 couples a deterministic gate-file emission to a monolithic, schema-less, multi-quadrant LLM synthesis call (originally an interactive skill, now a fixed-prompt `call_claude()`), so any prompt fragility or non-termination in that single call starves Phase 10 of its inputs and takes the entire closed-loop feature offline; the controllable technical fix is to drop Phase 9 LLM synthesis and have Phase 10 render the gate file deterministically from phase_4, optionally invoking per-quadrant, StructuredOutput-bounded LLM calls only for the human-readable summary.

8. **Why 8**: Why was role-creation reactive rather than coverage-mapped? Because the management system lacks a documented "oversight coverage matrix" (artifact class × oversight discipline × owner × cadence) that would make uncovered cells visible — without the matrix, gaps are invisible until an incident exposes them.  *new_insight*: No oversight coverage matrix means non-detection is the default state for any artifact class not yet incident-implicated.
9. **Why 9**: Why is there no oversight coverage matrix? Because the escalation-ladder doctrine (text → soft → hard → architectural) is defined per-instruction, not per-artifact-class — the management system optimizes for hardening individual rules rather than ensuring every artifact class has at least one detection mechanism at some rung.  *new_insight*: The escalation ladder is a per-instruction tool, not a per-surface coverage tool; applying it per-artifact-class would force minimum-detection guarantees.
10. **Why 10**: Why is the doctrine per-instruction rather than per-surface? Because the governance feedback loop (`feedback_*.md`, `escalation_log.yaml`, monthly review) ingests failures keyed by instruction/rule, not by artifact-class or pipeline-stage, so the data model itself cannot surface "this whole class lacks detection" findings.  *new_insight*: The governance telemetry schema constrains what management can see — keyed-by-instruction telemetry hides class-level coverage gaps.
11. **Why 11**: Why does the telemetry schema key on instructions? Because `metrics.jsonl` + `feedback_*.md` were designed to track CLAUDE.md rule adherence (the original failure domain) and were never refactored when skill pipelines, closed-loop auto-dispatch, and external side-effects (email, portal) were added to the ecosystem — schema lagged capability.  *new_insight*: Telemetry schema lag behind capability growth — each new architectural surface (closed-loop, side-effects) needs a corresponding telemetry dimension or it becomes invisible to governance.
12. **Why 12**: Why does telemetry schema lag capability? Because the management system has no policy that "adding a new architectural surface (side-effecting skill, new artifact class, new closed-loop) requires a paired telemetry & oversight-matrix update before merge" — capability changes ship without governance changes.  *new_insight*: ROOT (controllable) — no "capability-change ⇒ governance-change" coupling policy. Without this coupling, every new surface becomes a non-detection zone by default until an incident creates the feedback. This is the deepest controllable cause at the management-system level — fixing it (require a governance-update receipt for any new architectural surface, gating merge on coverage-matrix update + telemetry dimension + owner assignment + minimum detection rung) eliminates the class of "silent surface" that allowed Phase 9 to ship without an E2E gate, criticality tier, or detection mechanism.

**Q4 root**: No "capability-change requires paired governance-change" coupling policy: when a new architectural surface is added (closed-loop auto-dispatch, side-effecting skill stage, new artifact class), the management system does not require a paired update to the oversight coverage matrix, telemetry schema, owner assignment, and minimum detection rung before the change can land — so every new surface (including Phase 9's LLM-synthesis stage with email/portal side-effects) defaults to a non-detection zone until an incident exposes it.

## Wiki Ingest Drafts

The following insights are candidates for ingestion into the personal wiki:

1. **Closed-loop self-validator anti-pattern** (new concept page candidate) — When a validator's pass criterion depends on the artifact it must produce (e.g., Phase 9 LLM synthesis must produce plan.md to validate the closed-loop pipeline that contains Phase 9), the validator cannot bootstrap. The pattern recurs across automotive (jidoka), aviation (cockpit checklists), and broadcast (live B-roll fallback). Resolution: separate deterministic emission (the gate's machine contract) from optional LLM-synthesized human narrative; isolate any LLM use to bounded, schema-constrained, per-quadrant calls with deterministic fallback. Cross-link: `concepts/claude-agent-sdk-patterns.md`, `concepts/long-running-progress-heartbeat.md`.

2. **Interactive-skill-on-autonomous-surface failure class** (new concept page candidate) — Skills that prompt for human input deadlock when invoked from headless / batch / SDK auto-dispatch contexts. Mitigation requires (a) typed `execution_mode` metadata at the skill registry layer, (b) consumer-side mechanical gate at the skill-loading layer (not per-surface dispatcher), (c) behavioral conformance test that headlessly invokes autonomous-tagged skills and fails on `AskUserQuestion` emission. Cross-link: `concepts/instruction-failure-escalation-ladder.md`, `concepts/three-tier-lesson-learning.md`.

3. **Capability-change ⇒ governance-change coupling policy** (new concept page candidate) — When a new architectural surface is added (closed-loop, side-effecting skill stage, new artifact class), governance updates (oversight-coverage-matrix row, telemetry dimension, owner assignment, minimum detection rung) must land in the same change. Without coupling, every new surface becomes a non-detection zone by default. Implementable as a pre-merge hard gate (`hook-capability-governance-coupling.py`) over regulated paths. Cross-link: `concepts/silent-staleness.md`, `concepts/degraded-emission-with-warning.md`.

4. **Pattern-transplant scope-mismatch hazard** (lesson learned candidate) — Copying a successful LLM-call pattern (Phase 7) into a new stage (Phase 9) on the assumption "same plumbing → same reliability" smuggles in scope mismatch when the new task's payload, output schema, or task complexity is materially larger. Requires a contract-redesign step at every transplant: derive new acceptance criteria, write the post-condition validator, define the StructuredOutput schema. Without it, the transplant inherits the surface form but not the reliability envelope.

5. **Audit-input-truncation as recursive R13 instance** (lesson learned) — Producing an audit verdict on truncated/missing input would itself violate R13 (degraded emission with self-exonerating warning). The Phase 5 round 3 audit correctly refused to launder the missing Q4 input into a verdict. Future audit harnesses should add a producer-side jsonschema/pydantic precondition that refuses to deliver an audit packet missing either quadrant or with truncated evidence — the same content-shape predicate pattern used by R13.
